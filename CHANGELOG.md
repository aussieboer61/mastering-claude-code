# Changelog

- 2026-07-10 — Waiting on a worker that already died: a delegated agent that stops with "I'll continue when X completes" has ended its turn — if X finishes without re-invoking it (or it dies mid-report), nothing resumes it and no error surfaces; the finished work sits orphaned while the delegating session waits for hours. Discipline: past the expected window, poll the WORK STATE (output files, mtimes, commit log), not the worker; if the work is done and the worker is gone, take over inline rather than re-delegating. Mitigations: workers commit early/often, keep final reports terse (a long closing summary is a known death spot). Closing rule: every delegated deliverable lands somewhere visible (URL, rendered file, screenshot) — work only inside a repo you never open is indistinguishable from work that didn't happen. Added as new Ch 7 failure mode — Pillar(s): 7 (Async) — Files: published.md, narration.md, CHANGELOG.md, updates/week-08.md, updates/index.md
- 2026-07-10 — A rich store the assistant never opens: the quietest memory failure — everything is recorded, and the assistant builds a parallel system from scratch anyway because it never read the index before starting. Standing default: at the start of any build task, read the index, list the existing pieces that solve part of it, and design the smallest delta on those; about to author a parallel service/protocol = the signal to stop and extend instead. "A filing cabinet nobody opens is indistinguishable from an empty one." Added as new Ch 2 failure mode — Pillar(s): 2 (Memory) — Files: published.md, narration.md, CHANGELOG.md, updates/week-08.md, updates/index.md
- 2026-07-10 — Secret hygiene as a hook pair, not a rule: "never print credential values" is the canonical rule that keeps being broken as text. Prevention = `PreToolUse` (Bash) deny on any command piping a known secret file to a printing tool (allow-list checksums/moves/`source`); detection = `PostToolUse` scan of command output for credential shapes (token prefixes, private-key headers, `user:pass@` URLs) → block + treat-as-compromised/rotate. Pair with a sanctioned redacting reader so inspection stays legitimate. Added "A pair of hooks for secret hygiene" section + table row in Ch 4 — Pillar(s): 4 (Hooks) — Files: published.md, narration.md, CHANGELOG.md, updates/week-08.md, updates/index.md
- 2026-07-10 — Three behavioural defaults added to the Ch 1 table: (a) sanity-check results against known context — an implausible number (LAN-grade latency to another continent) is a signal to investigate, not a finding to report; (b) ask why the primary path is out FIRST — when running on a fallback, establish why the designed path isn't in use before engineering around the fallback; (c) the target is the deliverable — work built for a specific device/surface is done only when it works there; a browser/PoC stand-in is never the delivered answer unless explicitly accepted — Pillar(s): 1 (CLAUDE.md) — Files: published.md, narration.md, CHANGELOG.md, updates/week-08.md, updates/index.md
- 2026-07-10 — Commissioning a bespoke tool when an adopted one exists: tool discipline applies to what you ask the assistant to build. Custom dashboards rot — the hand-maintained list lags reality and the surface stops being opened. Ask "what would I honestly open every day"; adopt the maintained tool if one maps; build custom only when nothing covers the need AND the result derives live from an existing source of truth. Added as Ch 9 failure mode + narration paragraph — Pillar(s): 9 (Tool discipline) — Files: published.md, narration.md, CHANGELOG.md, updates/week-08.md, updates/index.md
- 2026-07-03 — Self-healing background runs: a background task can *guard* a risky live change, not just report it. Pair an outward-facing cutover (DNS flip, deploy, service repoint) with a watcher that polls for the desired outcome and auto-reverts if it doesn't materialise within a deadline — turning a babysit-required change into fire-and-forget with a safety net. Two disciplines: require a consistent success signal (two consecutive good readings, not one lucky one near the deadline), and make the rollback complete (unwind every side-effect, not just the DNS record). Added "Self-healing background runs" section to Ch 7 — Pillar(s): 7 (Async) — Files: published.md, narration.md, CHANGELOG.md, updates/week-07.md, updates/index.md
- 2026-06-18 — Find your hooks by auditing your own transcripts: cluster the moments you corrected or repeated yourself by *what went wrong* (not by task) and the top clusters are your hook backlog. Two high-value, non-path-blocking hooks documented: a `Stop` guard that refuses a "done/fixed/deployed" claim carrying no proof (blocks once, demands evidence or an honest "unverified"), and a `UserPromptSubmit` injector that re-states your hardest rules only when a message reads as a terse order/correction (silent otherwise). Framing: a rule the model reads can be rationalised past; a rule the harness enforces cannot — third repeat of a CLAUDE.md line = a hook waiting to be written. Added "What hooks are most useful for" rows + new "Finding which hooks are worth building" section in Ch 4 — Pillar(s): 4 (Hooks) — Files: published.md, narration.md, CHANGELOG.md, updates/week-06.md, updates/index.md
- 2026-06-06 — Trust operator on physical facts: when tool output (MAC tables, sensor counters, register reads) appears to contradict what the user says about their own physical environment, ask one clarifying question rather than assert they are wrong — remote data lacks on-site context. Added as new behavioural default row in published.md table and sentence in narration.md — Pillar(s): 1 (CLAUDE.md) — Files: published.md, narration.md
- 2026-06-05 — Session-stated rules stay binding: a method instruction or classification given early in a long session is a session-scoped rule, not a one-time answer; don't drift back to default heuristics as the session lengthens.
