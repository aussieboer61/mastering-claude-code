# Mastering Claude Code

## A practical guide for the curious — narrated edition

What you're listening to is a guide for people who've heard the name Claude Code, maybe tried it once or twice, and would like to get more out of it than the box gives you. There's also a written version with tables, diagrams, and a small bonus prompt at the end for an AI. If you prefer to read, look there. If you'd rather listen, stay here.

The point of this guide is straightforward. Most people who try Claude Code paste a question in, get an answer, and move on. That works, but it leaves nine-tenths of the tool on the shelf. This is for the moment you decide to pick up the rest of it.

Feel free to pass this along. The whole point is to make this less of a secret club and more of a shared toolkit.

## How this guide is laid out

There are four parts.

Part one is **Getting Started**. It assumes you've heard the name and not much else. It explains what Claude Code actually is, what you'll need, and walks you through a first session so the rest of the guide has somewhere to land.

Part two is **The Foundations**. Five chapters covering the five things you build once and then use every session forever. A standing letter to your assistant, called CLAUDE-dot-em-dee. A long-term knowledge store, called Memory. One-command procedures, called Skills. Rules your assistant cannot skip, called Hooks. And delegating to specialists, called Subagents.

Part three is **Composition**. Four chapters about combining the foundations into real workflows: state at the right scope, work that runs while you sleep, getting a second opinion automatically, and choosing tools well.

Part four is **Putting It Together**. It walks through one person's entire setup across all nine pillars, suggests an adoption order, and tells you when not to bother.

Listen straight through, or skip to the part that matches your current frustration. The chapters are independent.

# Part one — Getting started

## Chapter zero — What Claude Code actually is

Claude Code is a program you run in your terminal that talks to Claude, the AI. That's the headline. The detail is what makes it interesting.

When you use Claude through a web browser, you get a conversation window: you type, Claude replies, you type again. When you use Claude Code, you get the same conversation plus the ability for Claude to read files on your computer, run commands, edit files, and remember things between sessions — provided you set those things up. It's the difference between asking a friend for advice over the phone and giving them the keys to your kitchen.

Most people who try Claude in the browser run into the same wall: every conversation starts from zero. You re-explain your project, re-state your preferences, re-paste the same files. By the third or fourth conversation, you stop bothering with anything complicated and only use it for one-off questions.

Claude Code lets you build a working environment that remembers, enforces rules, and does work while you're not at the keyboard. The investment to set up each piece is small. The pay-off is that conversation number five doesn't start where conversation number one did. It starts wherever you left off, knowing what you've already established.

There are a few words you'll hear throughout this guide that are worth defining upfront. A *session* is one run of Claude Code in your terminal, from open to close — like one phone call. A *prompt* is a message you type, same as in the browser. *Context* is everything Claude can see right now — your prompts, its replies, anything it's read so far. Think of it as a finite scratch-pad. A *tool* is a specific action Claude can take, like read a file or run a command. An *agent*, sometimes called a sub-agent, is a second copy of Claude you can hand a task to, briefed cold, while your main session keeps doing other things. *Memory* is a folder of small text files Claude reads at the start of every session to remember what matters across conversations. A *skill* is a named procedure you've written down once that Claude can run on demand. And *CLAUDE-dot-em-dee* is a plain-text file at the root of your project that Claude reads at the start of every session — like a standing letter.

If those don't all click yet, don't worry. Each one has its own chapter.

To use Claude Code, you'll need a computer running Mac, Linux, or Windows (with the Linux subsystem). You'll need to be comfortable enough with the terminal to change directories and run a command. You'll need a Claude account — the Pro plan, around twenty dollars a month, is the usual starting point. And you'll need about thirty minutes to install and run a first session.

The official install instructions live at the Claude Code documentation. Once installed, a first session looks something like this. You open your terminal, change into a folder you care about, and type "claude". A prompt appears. You ask it what's in the folder. Claude reads the directory and tells you. You ask it to open a file and summarise a few entries. Claude opens the file and produces a summary. So far this is what the browser does too.

Now the difference. You tell Claude: "Save my preferences — I want concise answers, no preamble, bullet points only when they actually help." Claude, by default, can't do that. But with a memory store set up — which is what chapter two is about — it can write that to a small file and read it at the start of every future session. From that point on, every conversation knows that about you. You stop saying it.

That's the rhythm of the whole guide. Each pillar makes one piece of "I have to say this every time" stop happening.

## Chapter one — CLAUDE-dot-em-dee, a standing letter to your assistant

When you open Claude Code in a folder, it looks for a file called CLAUDE-dot-em-dee and reads it first — every single time, no matter what you're about to ask. Think of it as a one-page letter you've left on the desk for the assistant. *Here's what this place is, here are the rules of the house, here's where things live.* The assistant doesn't have to be told all of that again; it's already on the desk when they walk in.

Without it, every session starts at zero on your project. You re-explain the directory layout, re-state the conventions, re-mention the thing-that-must-not-be-touched. Halfway through the third re-explanation in a week, you realise you've been describing the same workspace over and over and the assistant still doesn't know it.

With it, the assistant arrives already oriented. The first minute of every session — the re-onboarding minute — disappears. Multiplied over a year, that's real time.

There are two places this file can live. One is user-wide — a file in your home directory's claude folder. That applies to every session, in every folder, wherever you happen to be. The other is project-specific — a file at the root of a particular project, applied only when you're working in that folder. You can use both. The user-wide file is for things that are true about *you* regardless of project: how you like to be talked to, your preferred file formats, any rule that should hold everywhere. The project file is for things specific to that project: the layout, the glossary, the gotchas. Both files are loaded together when you open the project; the project file extends the user file, it doesn't overwrite it.

What earns a place in CLAUDE-dot-em-dee? A two-line description of what the project is. The three or four commands you actually run. House conventions for naming and file organisation. A domain glossary — terms with specific meanings in this work. A handful of gotchas that have caught you before. Pointers to the rest of your setup — where memory lives, where skills live.

What does *not* earn a place? A full description of every directory. Every command anyone could possibly run. Things the code itself already documents. A textbook on the domain. Every conceivable edge case.

The discipline is **length**. Every word in this file costs you context budget on every session, whether or not today's task needs that word. Aim for fifty to a hundred and fifty lines. If you're over two hundred, start cutting.

A starting template is short. A header naming the project. One section on where things live. One on the commands you actually run. One on gotchas — say, "the staging server uses a different database than production, always confirm which one." And a final section of pointers — where memory lives, where skills are stored. Ten or twenty lines is a fine starting size. Grow it by noticing what you keep re-explaining, and adding those things one at a time.

A few things absolutely do not belong here. Secrets — no API keys, no passwords, no tokens. This file is plain text on disk, almost certainly checked into version control at some point. Current task state — "we're on step three of seven" is true today and stale tomorrow. That belongs in a plan file, which we'll come to in chapter six. And hard rules that must never be broken — if the consequence of skipping a rule is real, like a corrupted file or a leaked credential, that rule belongs in a Hook, chapter four, not in text the assistant *should* read but might miss when its attention is elsewhere.

The two most common failures are the five-thousand-word file and the wrong-scope file. The five-thousand-word file started at twenty lines and grew over a year into a reference manual. Now it eats context on every session and the assistant has learned to skim it because there's too much. Trim quarterly. The wrong-scope file happens when your communication preferences end up in one project's file because that's where you wrote them, and six months later in a different project the assistant doesn't know them. Put cross-project content in the user-wide file. Put project content in the project file.

And one more failure: writing this file like a README. A README is for humans new to the project — it explains, motivates, walks through installation. CLAUDE-dot-em-dee is for the assistant. Terse, directive, focused on what's needed to behave correctly. Conflating them produces a file that does neither job well.

One more thing earns a place in this file, and it's easy to miss because it isn't a fact about the project — it's a handful of standing instructions about *how* you want the assistant to behave. These matter more than any single project detail, because they shape every task rather than one. A few that prove their worth. Act when the work is authorised, and ask first only when a step is hard to reverse, genuinely undecided, or touches shared infrastructure — otherwise the assistant asks permission at every turn. Stay in lane: do the thing you asked, and treat a side-comment as a side-comment, not a licence to start a new project. Search before asking: look for a missing detail in memory and in the files before asking you to supply what's already on disk. Calibrate confidence: label what hasn't been verified, flag risks up front, and don't fold the moment you push back if the assistant is actually right. And evidence before "done": don't report a thing fixed until the symptom has actually been checked. Lead with the answer: when you ask a direct question — yes or no, who owes whom, how much — the answer belongs in the first sentence, with the supporting breakdown after it; a correct table you still have to read the answer out of hasn't actually answered the question. One more: read prior conversation mode before executing a destructive write — when the last few exchanges were evaluating an approach, a command like "run the migration" may still mean "test this" rather than "hit live now"; when the intent is ambiguous, one short confirmation beats a rollback. And one more: session-stated rules stay binding. A method instruction or classification given early in a long session is a session-scoped rule, not a one-time answer — don't drift back to default heuristics as the session lengthens; scan what was already established before applying defaults to any new instance of the same pattern. And one more: trust the operator on physical facts. When tool output — MAC address tables, sensor counters, register reads — appears to contradict what you say about your own physical environment, the assistant should ask one clarifying question rather than assert you are wrong. Remote data has context only the on-site person knows: a device that is not plugged in yet shows no traffic; equipment mid-installation has a state the sensor cannot explain. A close cousin: sanity-check results against known context. Before reporting a measurement as a finding, the assistant should check it against what's already known — geography, host state, time of day. A response time that looks like a local network when the server is on another continent isn't a result; it's a signal that something else answered. An implausible number deserves investigation before it's reported, not after you catch it. Another: when work is running on a fallback — a backup link, a secondary tool, a workaround — establish *why* the designed path isn't in use before engineering around the fallback. The reason usually changes the whole answer, and one question up front beats a scaffold built carefully around the wrong problem. And one more: the target is the deliverable. When something is built *for* a specific device or surface, done means working there. A stand-in — it runs in the browser, here's the same thing as a web page — is at most a temporary aid while the real target is being fixed, never the delivered answer unless you've explicitly accepted the downgrade. Two more, both about momentum. Finish the slice: when the plan is agreed and work remains, the assistant should work through to done rather than ending every increment with "want me to continue?" — and a standard follow-on step it already holds the keys for, the DNS record, the certificate, the service registration, is part of the work, not an optional extra to offer back. Ship the whole vertical slice, not the eighty percent that leaves you wiring the last mile yourself. And: the question outranks the instruction. When one message carries both an instruction and a question, the answer comes first — the question is the part blocking your next thought, while the instruction is delegation that can start in the background and report when it lands. Serialising you behind the delegated task wastes the one resource the whole setup exists to protect. Keep this list short, and phrased as what *to* do. A pile of "never do that" instructions produces an assistant that asks permission to breathe — and if your defaults have drifted to "ask before everything", that drift is the bug, not the safety net.

## Chapter two — Memory, knowledge that survives the session

Memory is a folder of small text files that Claude reads at the start of each session. Unlike CLAUDE-dot-em-dee, which loads in full every time, memory is indexed. Claude reads the index first, then opens only the files relevant to today's task. It's the difference between leaving a one-page letter on the desk and giving the assistant access to a filing cabinet with labelled drawers.

CLAUDE-dot-em-dee is short by design. Memory is where the bulk of your "what I'd otherwise have to re-explain" lives — your preferences, the lessons learned from past mistakes, the location of credentials, the quirks of a vendor's API, the decisions already made on each active project.

A memory store that starts small at week one feels indispensable by month six. The compounding is real. Every session adds a tiny bit of accumulated knowledge that all future sessions can draw on.

There are exactly four types of memory entry. *User* entries describe who you are and how you prefer to work — "I want concise answers and bullet points only when they help." *Feedback* entries capture a lesson learned from a mistake — "when I ask for a summary, don't add a 'next steps' section unless I asked." *Project* entries hold the running context of an ongoing effort — "the kitchen reno is in fit-out phase; cabinets arrive Friday; the certifier needs forty-eight hours' notice." *Reference* entries hold stable external facts you'd otherwise look up — "vendor X's quoting endpoint is slash api slash v3 slash quotes."

The two types you'll use most are feedback and project. Feedback compounds. Every mistake you bother to write down is a regression prevented across every future session. Project entries are where the running context of "where am I on the kitchen reno, the book draft, the consulting engagement" sits.

The store lives in your home directory, under a folder structure for whichever project Claude is working on. Inside that folder you'll find an index file, called MEMORY-dot-em-dee, and one short file per entry. The index is a table — one row per file, with a short description and a few tags. Claude reads the index first, then uses those tags to decide which other files to open. Each entry file has a tiny header at the top describing what it is, and a body — usually no more than a paragraph or two.

You don't need all four types on day one. The right order is: one user file, ten lines, who you are and how you like answers. Then one project file, twenty lines, for whatever's most active. Then a feedback file the first time the assistant does something you want it to stop doing — written the moment the friction happens, while it's fresh. Then a reference file the second time you find yourself looking up the same external fact. Once is fine; twice means it belongs in memory.

A few classic mistakes to avoid. Don't save things the codebase already knows. A note saying "the main function is on line forty-two" is duplicating what a search would tell you in less time than it took to write the note. Save decisions, conclusions, and external facts — not things the project can answer for itself.

Don't put plan-state into memory. "Currently in fit-out phase, cabinets arrive Friday" is true this week and wrong next week. Memory carries policies and standing facts; plans carry progress. We'll come back to this distinction in chapter six.

Don't let the index bloat. Every entry adds a tiny bit of overhead on every session. A pruned twenty-five-file index beats a sixty-file one with twenty stale entries. Skim quarterly.

And don't keep two files that say nearly the same thing. They'll drift apart. Pick one canonical home for each piece of knowledge. And one more, learned the hard way. Every time the assistant does something you don't like, the natural move is to write a feedback entry saying "don't do that." Each rule is reasonable on its own, but cumulatively they bias the assistant toward inaction. A session that opens with thirty "don't" rules and one default of "ask before doing anything" arrives at every task already half-paralysed. The symptom to watch for is the carve-out — each new feedback file becomes an exception to a default that's grown too cautious. When you see that pattern, the right move is to rewrite the default itself, not to stack another carve-out on top. Audit your feedback files every few months. Retire rules whose lesson is now obvious. Merge near-duplicates. And watch for the carve-out signal, because that's how you know the default has drifted. In practice the rewrite is small and the effect is fast. A default of "read and report, ask before acting" became "act when the work is authorised; ask only when a step is hard to reverse, undecided, or touches shared infrastructure", and the half-paralysed feeling lifted inside a single session.

And one related discipline, learned the same way. Every fact in memory should trace back to where it came from — something you said, a document on disk, or a decision the two of you made together. The hazard is the unsourced guess. The assistant meets a gap, fills it with something plausible, writes it down, and from then on every piece of work that draws on memory treats the guess as established fact. A fabricated detail doesn't sit still — it propagates through everything derived from it. So cite the source on each entry, and mark anything inferred as inferred. This bites hardest with anything written in your own voice: a single invented biographical detail, once in memory, will resurface in letters and documents you never thought to check.

And the quietest failure of all: a rich store the assistant never opens. Everything is recorded — the working pipeline, the component that already does ninety percent of what you just asked for — and the assistant builds a parallel system from scratch anyway, because it never read the index before starting. The cost isn't a wrong fact; it's a wasted session of duplicate work, and a second system to maintain forever. The fix is a standing default worth writing down: at the start of any build task, read the index, list the existing pieces that already solve part of it, and design the change as the smallest step on top of those. The moment the assistant is about to author a new service or pipeline that parallels one already in the store, that's the signal to stop and extend the existing one instead. A memory store pays rent only when it's consulted at the moment of decision — a filing cabinet nobody opens is indistinguishable from an empty one.

Here's how it plays out. A researcher is writing a doctoral thesis over eighteen months. She makes one user entry noting she prefers inline citations over footnotes and wants her claims flagged when evidence is thin. She makes one project entry describing her research question, the structure of her argument chapters, and the location of her bibliography database. Three months in, she adds a feedback entry: when she asks for "a more careful tone", Claude over-qualifies; the rule is "tighten the hedges, don't add new ones". A few weeks later, another: a particular statistical method was questioned by her supervisor, so only use it when the dataset meets a specific condition. And a reference entry holds her department's style guide — specifically, the edge cases she keeps re-looking-up. By month twelve, every session opens already knowing her preferred citation style, the structure of her argument, which statistical decisions are settled, and which writing patterns she's already corrected. The session begins at the level of a collaborator who's been alongside the project from the start, not a fresh assistant who needs re-onboarding.

## Chapter three — Skills, one-command procedures

A skill is a small file that bundles three things together. A name — the slash-command you type, like "slash style-pass". Instructions for what to do when called. And a description that tells Claude when to use this skill. Once you've written it, you can invoke it explicitly by typing its slash-command, or Claude can invoke it automatically when your prompt matches its description.

Without skills, multi-step procedures live in your head or in old transcripts. You re-explain the same sequence every session, the details vary each time, the assistant drifts from your intent. Skills convert "something I worked out once" into "something we repeat reliably". When Claude reads its skill library at the start of each session, it sees the procedures available and can reach for one without being told.

Physically, a skill is a folder containing a single file called SKILL-dot-em-dee. The folder name is the skill's name. The file has a short header — frontmatter, in the jargon — with the skill's name, description, and the list of tools the skill is allowed to use. Below the header is the body, which is the procedure itself. Write the body like you're briefing a colleague who will do exactly what you've written, no more, no less.

The most important field in the file is the description. It does two jobs at once. It tells *you*, and future readers, what the skill does. And it tells *Claude* when to reach for the skill without being asked. Vague descriptions like "useful for prose" never trigger automatically. The pattern that works is: state what the skill produces, then list the literal phrases you might type when you want it. Over-specify. The system under-triggers more often than it over-triggers.

Skills live in one of three places. User-scope, in your home claude folder under skills — available in every session, in every folder. Project-scope, in a claude folder at the root of a specific project — only available when you're in that folder. Plugin-scope, distributed as a package. A project skill with the same name as a user skill takes precedence. Useful when you want a generic skill but override it for one specific project.

What's worth writing as a skill? Apply the third-time rule. If you've explained the same multi-step procedure three times across sessions, it belongs in a skill. Some examples: a morning brief that pops the inbox, lists urgent items, and gives you a one-line health check on the systems you care about. A weekly review that summarises the week's activity and flags anything you said you'd do but haven't. A finishing-touches pass on writing — style checks, citation format, the usual cleanup. And a deploy or release procedure — document the exact sequence once, then type "slash deploy" thereafter.

To build one, draft the skill by hand or use Claude's own skill-creator skill, which interviews you about edge cases. Scope the tools tightly — only list what the skill genuinely needs. A prose skill doesn't need shell access. Test the skill against a realistic scenario, reading the full back-and-forth, not just the final output, and look for missed steps. Then tune the description until Claude triggers it on the phrases you actually use.

There's a second kind of skill worth knowing about. The ones we've described are procedures you invoke — a deploy, a style pass, a sequence you'd re-type. But some skills encode *how to approach a class of work*, and their whole value is that they fire before the work, not during it. A brainstorming skill that runs before you build any new feature, to pin down what's actually wanted. A test-first skill that insists the test exists before the code. A debugging skill that makes you find the cause before proposing a fix. Call these process skills. The catch is that the discipline only works if it's consulted before you charge in — a process skill you remember only after you've picked an approach has done nothing. And you don't have to write these yourself; they increasingly ship as installable plugin packs, which is the plugin scope from a moment ago, used in earnest. Reading a few well-made ones is also the fastest way to learn what a good skill looks like from the inside.

The failures to watch for: vague descriptions that never trigger; skills that wrap a single command (just type the command); skills that should be agents (if a skill runs for forty-five minutes, fans out in parallel, or needs to ask the user mid-procedure, it's not a skill, it's an agent); and hardcoded paths in user-scope skills, which break the moment the skill runs in a different project.

If you're trying to decide between a skill, a hook, and a subagent — chapters four and five — here's the rule. A hook fires automatically on a system event, like a session start or a file write. A skill is invoked on demand by the user, or pro-actively by Claude based on the description. A subagent is for work that's long, parallel, or needs a specialist with its own tool kit. We'll come to hooks and subagents next.

## Chapter four — Hooks, rules your assistant cannot skip

A hook is a small script that fires automatically at specific moments in a session. Before any file is written. After a tool runs. When the session starts. When the assistant finishes a turn. The hook runs in your shell, as you, deterministically. If it exits cleanly, the event continues. If it exits with an error, the event is blocked.

Hooks are the difference between *telling* the assistant to follow a rule and *making* the rule happen. Text in CLAUDE-dot-em-dee can be skimmed when context is crowded. A hook cannot be skimmed. It runs every time.

Some rules need to hold every time, no exceptions. "Never write to the production database directly." "This folder of documents is frozen — no edits without approval." "Send me a notification when a long job finishes." These are not preferences; they're invariants. Putting them in CLAUDE-dot-em-dee gives you the appearance of safety. Putting them in hooks gives you the substance.

Hooks fire on specific events. Session Start fires once, immediately after the session opens. User Prompt Submit fires each time you submit a message, before Claude processes it. Pre Tool Use fires before any tool call, and you can scope it to specific tools — so you can have a hook that fires only on file writes, or only on shell commands. Post Tool Use fires after a tool call returns. Stop fires when the assistant finishes a turn. Sub-agent Stop fires when a subagent finishes its turn. And Notification fires when Claude emits a user-facing notification.

The most useful uses of hooks, in practice. Injecting today's date and a one-line status at session start, so the assistant arrives knowing what day it is. Blocking writes to protected paths — frozen files literally cannot be edited. Sending a notification when a long task finishes — you get pinged when something's done, without needing to watch the terminal. And appending an audit log entry after a sensitive command, so a record exists regardless of what the assistant decided.

The cheapest useful hook is a Session Start script that injects today's date. Claude doesn't otherwise have a reliable view of the current date unless you tell it. The script is three or four lines of shell that emit a small JSON payload with today's date. You save it as a file, make it executable, and wire it up in your claude settings file. From then on, every session opens already knowing what day it is.

A more useful pattern is a Pre Tool Use hook on file Write and Edit that checks the file path and blocks the operation if it points at a frozen directory. The hook reads the tool call from standard input, pulls out the target path, checks it against a list of frozen directories, and either lets the write pass through or returns a "blocked" response with a friendly reason. The assistant cannot edit frozen documents, not because it was *told* not to, but because the harness physically stops the write.

One rule deserves this treatment more than any other: never print credential values. It's the canonical example of a rule that keeps being broken as text. It sits in the standing instructions, the assistant means well, and then one day it prints a config file to check a hostname and a live API token rides along into the transcript. The transcript is a document — logged, synced, sometimes shared — so the moment a value is printed, it's exposed. The enforcement is a pair of hooks on shell commands. Prevention first: a Pre Tool Use hook that denies any command piping a known secret file — key files, credential stores, token caches — to a printing tool, while allowing the safe handlers: checksums, file moves, and loading values into the environment without displaying them. Then detection: a Post Tool Use hook that scans command *output* for credential shapes — provider token prefixes, private-key headers, addresses with a password embedded in them — and on a hit, blocks with an instruction to treat the value as compromised and rotate it rather than re-run the command. Pair them with a sanctioned reader: a small script that prints any file with secret-shaped values replaced by a fingerprint, so there's always a legitimate way to inspect a config and the hook never becomes an obstacle to route around. Prevention makes the mistake structurally impossible; detection makes the residual failure loud instead of silent. Neither depends on the assistant remembering anything.

A few classic failure modes. Encoding a real rule as text instead of a hook. "Never write to the production database" in CLAUDE-dot-em-dee is a wish. The same rule in a Pre Tool Use hook is enforced. The test: would it matter if this rule were skipped *once*? If yes, make it a hook.

Overbroad matchers — a hook that blocks all writes to a directory will also block writes the assistant legitimately needs to make. Be precise; include a bypass mechanism, like a one-shot unlock file, so legitimate work isn't permanently stuck.

Context-window bloat from Session Start hooks — a hook that injects a giant document on every session start eats budget that could go toward actual work. Keep injected context to a few lines.

And host-specific hooks without environment checks — a hook that calls a tool only available on one machine will fail silently or noisily on every other machine where the same settings file applies. Either check for the dependency inside the hook, or scope the hook to the project that needs it.

One more, easy to hit once you have a few hooks: shared state that leaks across sessions. The moment a hook writes a file — a flag that unlocks a guard for one operation, a lock that says this session owns the inbox, a marker that stops work being done twice — that file is shared by every session running at once. Give it a fixed name and two sessions collide: a one-shot unlock you granted in one terminal satisfies a guard in another, or a flag left behind by a session that has since died blocks a live one. Put the session's own identifier in the filename, so a bypass stays where you granted it. And one that hides in plain sight: a guard that fires once and then goes quiet for the rest of the session. Blocking only the first offence feels tidy — it isn't nagging — but the offence it catches is rarely a one-off. A session that claims something done without proof once tends to do it again as it lengthens, and those later claims are exactly the ones that slip through, because the guard already did its job and fell silent. Read such a guard's own log back across several long sessions and the shape is unmistakable: one early block, then a cascade it never stopped. The fix is a small counter kept in the state file — re-arm on the first offence and every fifth after it, first, fifth, tenth — so the guard keeps its bite through the long sessions where discipline slips the most. A hook that can only interrupt once is a hook that stops working the moment it's needed twice.

So which hooks should you actually build? The ones you need are hiding in your own history — in the moments you had to correct the same thing twice. The transcripts of your past sessions are on disk. Read back over a month of them, or point the assistant at them and ask it to cluster the moments where you pushed back or repeated yourself — grouped by what went wrong, not by what the task was. The pattern that comes back is rarely that it can't code. It's behavioural and repetitive: it claimed something was done when it wasn't, did work you didn't ask for, kept asking permission you'd already given, leaned on a stale note instead of checking the live system. Those clusters are your hook backlog, ranked by how often they cost you.

Two of the most valuable hooks come straight out of that exercise, and neither one blocks a file path. The first is a Stop guard that refuses an unproven claim. It reads the assistant's final message, and if that message asserts something is done or fixed or deployed but shows no evidence — no command output, no status code, no commit hash, no figure — it blocks and asks for proof, or for an honest admission that it couldn't verify. "It works" stops being something the assistant can say for free. And a warning that comes back later in this chapter applies to exactly this hook: a guard like it must not fire only once per session. The second is a prompt-time injector that fires only on heat. Most of the time it stays silent. But when your message reads as a terse order or a correction — just do it, I already told you, stop asking — it quietly prepends your hardest-won rules to that one turn. The discipline arrives exactly where the record shows it tends to slip, and costs nothing on the calm turns.

Why a hook and not another line in your standing instructions? Because a rule the model merely reads is one it can rationalise past under pressure, and a rule the harness enforces is one it cannot. If you find yourself adding the same instruction for the third time, that's not a documentation gap. It's a hook waiting to be written.

The three-way choice between hook, CLAUDE-dot-em-dee, and skill comes down to this. A rule the model *must* obey, every time — that's a hook. A standing piece of context the model *should* have — that's CLAUDE-dot-em-dee. A procedure the user invokes on demand — that's a skill. A reminder belongs in CLAUDE-dot-em-dee. An invariant belongs in a hook. A repeated action belongs in a skill.

## Chapter five — Subagents, delegate the heavy lifting

A subagent is a second copy of Claude you can hand a specific task. It arrives with no memory of your conversation, carries only the tools you've permitted it, and may run on a different model than your main session. You write its briefing once and store it as a file; from then on, your main session can call it by name whenever that kind of work appears.

The main session orchestrates — it holds the plan and the judgement calls. The subagent executes a bounded task and reports back. The separation isn't a courtesy; it's the structural reason subagents are useful at all.

There are three reasons to delegate. The first is **context hygiene**. Every tool call that returns a large result — a long log file, a directory full of documents, a database query — lands in your context window and stays there. A subagent does the heavy reading and returns only the distilled answer. Your main session holds reasoning; the subagent absorbs the noise. The second is **parallelism**. Three independent research questions can run at the same time, finishing in the time the slowest takes. A single session can't do that for itself. The third is **specialisation**. An agent briefed as a domain specialist is less likely to drift than a general session handling one more thing at the bottom of a long thread. The brief is the agent's whole world; that focus is a correctness advantage as much as a speed one.

A subagent is a single Markdown file, in your home claude directory under "agents". The frontmatter at the top defines the agent — its name, its description (which Claude reads to decide when to invoke it), the list of tools it's allowed to use, and the model it should run on. The body below is its persistent briefing — its role, the conventions it follows, the rules it applies when it hits something unexpected.

Tool allowlists matter. A research agent that reads files and runs searches doesn't need Write or shell access. A formatting agent doesn't need Bash. Tight allowlists mean three things: less risk if the agent misinterprets its brief, faster invocation, and clearer intent — the allowlist documents what the agent is supposed to touch.

The model choice matters too. Haiku is for high-volume mechanical work — extracting structured data, formatting, simple classification. Sonnet is for standard professional work where judgement matters. Opus is for reasoning-heavy work where getting it wrong is expensive — legal analysis, architecture, subtle debugging. Running ten parallel Haiku agents to summarise ten documents costs a fraction of a single Opus call and often produces equivalent results.

The agent's definition is permanent — it lives in the file. The **brief** is what you pass at invocation time, the specific task right now. A good brief looks like this: one sentence saying what "done" looks like. Exact file paths or data — not "the report" but the actual path. Format and destination — "write findings to a specific file as a structured list." Things not to do, word limits, style. And "if stuck, report back rather than guess." A vague brief produces vague work. The agent can't ask you a clarifying question mid-task unless designed to do so, so the brief must be self-contained.

When should you delegate, and when should you do it yourself? A task that fits in three lines, with a small output — do it inline. Output that will be large and you don't want cluttering context — subagent. Task that's one of several independent things — subagent in parallel. The work has a specialist angle you've defined an agent for — subagent. Mechanical, repeatable sequence with no judgement — that's probably a skill, not an agent. The break-even point is roughly thirty to sixty seconds of careful effort. Below that, do it yourself; agent invocation overhead exceeds the gain.

The most powerful pattern is parallel fan-out. When you have N independent subtasks of the same kind — summarise these ten documents, draft these five sections, check these eight references — spawn N agents in one go. They run simultaneously and finish in the time the slowest one takes. There are two rules. First, validate the outputs — read what the agents actually produced before treating it as done. Confident-sounding output isn't always correct output. Second, watch for sibling collisions. If the agents share a design space — they must produce distinct domains, consistent naming, coordinated structure — flat fan-out without coordination causes overlap. Two agents picking the same obvious example independently. The fix is a manager agent that assigns scope to each worker before any of them begin.

A few classic failures. The terse brief — "fix the formatting issues in the report" produces shallow work because the agent must guess what the standard is, where the issues are, and what good looks like. Output quality reflects brief quality. Over-delegation of trivial tasks — spawning an agent to rename a variable is slower than doing it inline. Serial fan-out — launching agents one at a time when they're independent wastes the whole point of the pattern. Trusting the summary — an agent that says "done, all three sections updated" might have updated two correctly and one incorrectly. Read the diff yourself. And building a subagent for what should be a skill — if the "agent" is a fixed sequence of commands with no judgement calls, it's a skill. Subagent equals packaged expert; skill equals packaged procedure.

## Chapter six — The persistence ladder, state at the right scope

Information you produce in a session lives at one of four scopes. Within a single response. Across this conversation. Across all conversations. Or across parallel branches of work. Each scope has a natural home, and each carries overhead proportional to how durable it is. The discipline is picking the smallest scope that genuinely fits.

When you store the wrong thing at the wrong scope, two failures occur in opposite directions. Putting plan-state into memory means every future session loads stale progress into context. Putting durable conventions into a session-only todo list means they evaporate by tomorrow. Neither failure announces itself loudly — you just gradually notice things drifting.

The lowest rung is **todos**, inside a single session. Claude has a built-in todo tool that maintains a checklist within the current conversation. Use it freely for the sequence of steps in front of you right now — call the plumber, draft the variation notice, confirm the inspection. When the session ends, the list ends. That's the design. Don't treat a todo list as a deliverable, a decision log, or anything Claude should remember next time.

The next rung up is a **plan**, which is what a todo list becomes when the work spans more than one session, requires a decision between phases, or has dependencies worth reasoning about in advance. A todo answers "what am I doing right now in this session"; a plan answers "what are the phases, in what order, and what does done mean for each". Save plans to a known location — under a "plans" folder in your home claude directory is the convention. Not in temp, not in a session buffer. The plan is an artefact: it lives as long as the work lives, and gets archived when the work is done.

Above that is **memory** itself, which we covered in chapter two. Memory carries policies and standing facts, not progress. The test is one question: "Will I want this in a session that has nothing to do with the current work?" If yes, it's memory. If only while this project is active, it's a plan or project file. The classic mistake is writing plan-state to memory — "currently in fit-out phase, cabinets arrive Friday" — which is wrong by next week and poisons every session that loads it.

The fourth rung is **worktrees**, or sandboxes. If your work is code, a worktree is a second checked-out copy on a different branch. You can experiment without touching the mainline; merge or discard the result. If your work is documents, the equivalent is a sandbox copy — a separate folder holding an experimental variant while the authoritative version stays untouched. Use worktrees when you'd otherwise risk clobbering or contaminating finished work. The overhead is real, so don't reach for one for a small in-place edit.

The default walk for any new piece of work goes like this. Start with todos. Let Claude track the steps. If the work finishes in one session, you never climb higher. Promote to a plan when the work spans sessions or needs a sign-off between phases. Persist to memory only when a decision has proven to generalise beyond this project. Branch a worktree when you'd contaminate the mainline by working in place.

The failures match the rungs. Promoting todos to permanent documents — if a list is worth saving, rewrite it as a proper plan with phases and success criteria. Plans for one-session work — a five-phase plan for "call the plumber" is overhead. Plan-state in memory — already covered. And a worktree for a trivial edit — the overhead of a second branch is only justified when isolation is genuinely needed. One more failure hides where two tools meet: two retention windows, with the shorter one upstream. An archive job that sweeps everything older than ninety days sits downstream of a tool whose built-in cleanup deletes the same material at thirty. Every item is purged two months before the sweeper would have collected it — the archive stays nearly empty indefinitely, and nothing errors, because deletion is a default and defaults don't announce themselves. When you build anything archival, list every cleanup already acting on the source — the harness's own retention setting first — and make the upstream window longer than the downstream sweep, or switch it off. The audit is a single check: if the oldest surviving item is exactly as old as some tool's default retention period, that tool is winning.

Here's a homeowner's renovation. Month one: she uses Claude's in-session todos to track each day's site work — confirm skip-bin delivery, photograph existing walls, call the certifier. The lists do their job and are gone by evening. Month two: scope has grown. Three phases, two hold points for sign-off, a materials schedule tied to lead times. She writes a plan to her plans folder with labelled phases. Each session begins by reading the plan. Month four: she notices she's restating the same conventions at the start of every session — variation notices follow a specific format, trades get forty-eight hours' written notice before any schedule change. These are policies, not plan-state. She moves them into memory. When the builder suggests moving the island bench six hundred millimetres — a change that might void the tile order — she doesn't edit the confirmed plan. She makes a sandbox copy and works through the implications there. The change doesn't survive scrutiny. She discards the sandbox. The authoritative plan is untouched.

## Chapter seven — Async, work that runs while you sleep

You're not always at the keyboard. A mature setup builds that assumption in rather than hoping you'll be there. The async layer has three pieces. Outbound notifications, where something pings you when a task completes. An inbound inbox, where you can drop a message in from your phone that the next session picks up. And background runs, where long jobs don't block your main session.

Long-running work without notifications forces a choice: park yourself at the terminal or check back hours later to find it either finished cold or errored on step two. An inbound inbox solves the symmetric problem: thoughts that occur away from the machine evaporate before the next session starts unless you have somewhere to put them. Without the async layer, Claude Code is a synchronous command prompt. With it, it's a delegatable agent that works while you move and sleep.

The simplest outbound notification is a one-line shell wrapper around your preferred push channel — Telegram, Pushover, an ntfy.sh topic, an email gateway, anything you actually check. You give it a message, it pushes. Keep the messages substantive. "Done" is useless. "Done — zero failures, output at this path" is actionable. A notification is a state-transition signal — task complete, decision needed, long job starting — not activity noise.

The inbound inbox is a persistent queue you can append to from anywhere. Your phone, another machine, a web form. The next Claude session pops messages off it before starting other work. The queue must be persistent — not in-memory — capped so it doesn't grow forever, and readable non-destructively so you can inspect it without consuming entries. The assistant's job at session start is to check inbox, act on any messages, then proceed.

Background runs are the third piece. Claude's Bash tool and Agent tool both accept a "run in background" flag. Use it for any subprocess that will take more than a few seconds. The main session doesn't block; you get notified when it finishes. Pair background runs with the Monitor tool, which streams events from a running background process. Each line of output becomes a notification, so you can watch a build log in real time without blocking the main session.

A background run doesn't have to just report — it can guard. When you make an outward-facing change that's risky to leave broken — flipping DNS to a new host, cutting over a deploy, repointing a live service — pair the change with a background watcher that verifies the outcome and reverts automatically if it doesn't materialise. The shape is: make the change, then poll for success, and require the signal to hold before trusting it. If success arrives, the watcher exits clean. If the deadline passes first, the watcher runs the rollback itself and reports what it did. This turns a cutover you'd normally babysit into fire-and-forget with a safety net: the worst case is a bounded window, then automatic recovery, rather than a service left down until you happen to notice. Two disciplines make it trustworthy. Require the success signal to be consistent — two consecutive good readings, not a single lucky one near the deadline. And make the rollback complete: restoring the DNS record isn't enough if the cutover also registered the hostname somewhere that now blackholes it. The rollback has to unwind every side-effect, or the safe path is still broken.

There are three pacing primitives. For fixed schedules — a nightly build, a weekly review — use cron. For self-paced loops, where the assistant polls until some state changes, use a "slash loop" skill or equivalent. For in-session waits where the assistant pauses for some known interval, use Claude's Schedule Wakeup tool. For those in-session waits, watch the prompt cache window. Roughly: under five minutes, the cache stays warm and re-entry is cheap. From five minutes to an hour, you pay a cache miss. Longer than an hour, the cost is irrelevant. The awkward middle is a twelve-minute sleep on an eight-minute job — you cross the cache boundary for no reason. Either stay short or commit to a real pause.

Here's a nightly bake. A solo game developer runs a nightly asset bake — forty level maps, sprite atlases, lighting pass, packaged build. Three to five hours, no reason to sit and watch. She fires it at ten pm. The opening notification reads: "Starting nightly bake, forty levels queued, estimated three to four hours." The export subprocess runs in the background. The Monitor tool streams the log. She drops two notes from her phone before bed: "If level eighteen fails the lighting pass again, skip and flag it, don't block the bake." And: "Jump-height values in levels twelve through fifteen were rebalanced today; note that in the manifest." At the midpoint, she gets a checkpoint: "Twenty of forty baked, no failures. Atlas export next." At one am, level eighteen's lighting pass fails. The assistant checks the inbox, finds the standing instruction, skips, flags, continues. No one is woken. At five-thirty am, the bake completes: "Done — thirty-nine of forty packaged, level eighteen flagged, jump-height note added to manifest." She reads it over breakfast.

When to ping and when to stay quiet. Ping on completion of anything that took more than a few minutes. Ping on errors you can't resolve from standing instructions. Ping at natural checkpoints in multi-phase work. Ping with a heads-up for genuinely long jobs starting. *Don't* ping on routine intra-phase progress. *Don't* ping on decisions you can resolve yourself. *Don't* ping on anything that will finish before you could read the notification. A quiet channel is a trusted channel. One substantive, well-timed notification beats five noisy ones.

The failure modes. Alert fatigue — notifications on every tool call mean you'll mute the channel within a day. Silent completion — opposite failure, the job finishes at two am, you find out at eight am, half the day already gone. Polling loops that thrash the cache — a sleep-ten-seconds loop for a twenty-minute process generates calls every ten seconds for no reason. Inbox messages without context — "fix the thing" is useless by morning, each message needs enough context for a cold session to act. And one more: the async harness silencing the human. When a background task is running, the assistant's bias toward "do the work, report back" can cause it to queue your new prompts behind the running task. A new message from you is always an interrupt. If the message has a question, answer it now.

And one last failure mode, learned the expensive way: waiting on a worker that already died. A delegated agent that stops with "I'll continue when the build completes" has ended its turn. If the thing it was waiting on finishes without waking it up again — or the agent dies partway through its final report — nothing resumes it, and no error surfaces anywhere. The finished work sits orphaned on disk while the session that delegated it keeps waiting for a worker that will never land, sometimes for hours. The discipline: when a delegated deliverable is well past its expected window with no word, poll the work state, not the worker. Look at the output files and their timestamps; look at the commit log in the target repository. If the work is done and the worker is gone, take over and finish it directly — don't send another messenger after the first one. Two habits make this failure cheaper when it happens. Have long-running workers commit early and often, because work saved before a death survives it. And keep their final reports short — a long closing summary is a known place for a worker to die with everything undelivered. One closing rule for anything delegated: the deliverable should land somewhere you can actually see — a link you can open, a rendered file, a picture in the reply. Work that exists only inside a repository you never open is indistinguishable from work that didn't happen.

## Chapter eight — Multi-agent patterns, a second pair of eyes

A single agent is fast but biased toward its first plausible answer. Once it commits, every subsequent step reinforces the direction rather than questioning it. A second agent briefed cold — meaning it hasn't seen any of the first agent's reasoning — can't anchor on a framing it never received. That structural independence is the point. The cost is roughly linear in tokens. The quality lift, especially where a confident wrong answer is worse than a flagged uncertainty, is not.

A reviewer who didn't write the plan sees gaps the author assumed away. A second agent briefed cold is in exactly that position. This matters most for hard-to-reverse decisions and for judgement calls where "looks right" isn't the same as "is right". Cheap verification you can do yourself — running tests, checking outputs, eyeballing a diff — doesn't need a second agent. When verification requires a perspective you cannot structurally take, add an agent.

There are five useful patterns worth knowing.

The **critic loop**. One agent produces; a second agent, briefed cold, critiques against an explicit rubric; the producer incorporates the critique; the critic re-checks. Set a ceiling — two or three rounds — before you start. The critic must receive only the output, never the producer's reasoning. A critic who sees intent evaluates effort, not result.

The **planner-executor split**. A reasoning-heavy planner — the most capable model available — produces a structured plan. A cheaper executor carries out each step. Different bottlenecks: planning is reasoning-intensive and mistakes compound; execution is often mechanical once the plan is sound. Running an expensive model for planning and a cheaper one for execution usually beats running either end-to-end.

The **second opinion**. For a single high-stakes judgement — is this clause ambiguous, is this schedule feasible — spawn a second agent with the same inputs and a brief that references nothing the first produced. Agreement raises confidence. Disagreement locates exactly where the question is genuinely hard.

**Parallel fan-out with synthesis**. N agents on N independent questions, then a synthesiser agent that produces a unified result. The synthesiser is briefed to reconcile contradictions and flag gaps, not just stitch outputs together.

**Build-and-grade**. A producer drafts; a grader scores against an explicit rubric; producer iterates; grader re-evaluates. The score makes convergence visible — you can see whether rounds are improving the work or spinning in place.

Those five patterns are things you drive by hand — you spawn an agent, read what it says, decide, spawn the next. That's right while you're still feeling out a problem. But once the shape of the work has settled — fan out across this list, verify each finding, keep the survivors — you can hand the control flow itself to a script. A workflow is that script: the loop, the fan-out, and the branching written down as deterministic steps, with agents as the unit of work inside. A small vocabulary covers most of it. A pipeline runs each item through every stage on its own, so one finding can be at the verify stage while another is still being discovered — nothing waits for the slowest item at each step, and that's the sensible default. A barrier is the opposite: collect every result from a stage before the next one starts, which is right only when the next stage genuinely needs the whole set, to merge duplicates or to stop early if the count came back zero. Loop-until-dry keeps spawning finders until a round or two turns up nothing new — the honest way to size a search whose answer you can't know in advance. Adversarial verify spawns independent sceptics for each finding, briefed to refute it, and keeps only what survives a majority; that's the step that stops a confident wrong result from reaching you. And a judge panel generates several independent attempts, scores them, and synthesises from the winner. Reach for a workflow when the work is large and its shape is known. When the shape isn't known yet, do the reverse — scout by hand, discover the work-list, then hand it over. Automating a guess only produces a great deal of confident guessing. And one posture worth naming on its own: exhaustive mode, the deliberate choice to spend tokens buying confidence, fanning out wider and verifying harder than the answer strictly needs, because this particular decision is expensive to get wrong. It's the right call for the irreversible and the costly, and the wrong default for a quick lookup.

Here's a wedding plan. An event planner has drafted a two-hundred-person wedding plan: venue logistics, catering, run-sheet from access to carriages, three vendor contracts, a risk register. A single end-to-end review will miss cross-domain conflicts. The food-safety reviewer won't catch the clause in the venue contract that prohibits outside catering without a supplement. Instead, parallel fan-out to five specialist critics, each holding the full plan but briefed cold on a specific domain: venue logistics, catering, run-sheet, contracts (instructed to cross-reference the others), and contingency (instructed to assume the three most likely failure modes). Each writes findings to a named output file. Then a synthesis agent, briefed cold on the five critique files, looks for contradictions, repeated flags — meaning real structural problems, not quirks — and a ranked list of blocking items by impact on the date. The pattern earns its cost because the five sub-tasks are genuinely independent, domain-specific bias would cause a generalist to miss problems systematically, and the event date can't move. Where a confident wrong answer means a ruined wedding instead of an extra revision, the redundancy is cheap.

The failure modes. Critic loops that never converge — no stop rule means the critic always finds something. Set a ceiling. Accept good-enough after three rounds. Planners that over-specify — specifying every micro-step leaves the executor no room to respond to what it actually finds. Plans should specify outcomes per step, not a rigid command sequence. Second opinions that share the first agent's conclusion — "Agent A concluded X, what do you think?" is a request for validation, not a second opinion. The second agent must receive the same raw inputs as the first, nothing more. Parallel fan-out for dependent tasks — running B in parallel with A when B needs A's output produces error-embedded results that look complete and aren't. Sibling collisions in flat fan-out — N parallel agents picking the same obvious example domain independently. The fix is a manager agent that assigns distinct scope before workers begin. A nastier cousin: parallel sessions colliding in another failure's clothes. Two sessions working the same codebase on one machine rarely fail with "another session interfered" — they fail as something else. A test suite dies mid-run with what looks like an out-of-memory kill; the real cause is the sibling's test run tearing down the shared containers under it. A coordination registry is unreachable and fails open, so each session believes it holds the lane alone. A merge completes without a single conflict while silently dropping one branch's additions, because the other side rewrote the same files wholesale and auto-resolution took the rewrite. The disciplines: give each concurrent run its own namespace — test project names, ports, and a working copy each, a worktree per session, never a shared checkout; treat an unexplained kill signal as "check for a sibling" before debugging the code; and after any merge where the other side heavily refactored shared files, search the merged tree for your branch's key symbols before trusting it — a conflict-free merge is not a content-preserving merge. Critic monoculture — five critics all briefed with the same axis, all adversarial, all security-focused, all compliance-oriented, will collectively miss every failure outside that axis. At minimum, one critic should occupy the "use this as an actual operator would" angle, treating the work as a product to be used, not a specification to be verified. Orchestrating before you know the shape — a workflow is for structure you can already name; reach for one before you've scouted and you've only automated a guess. Silent caps — a workflow that quietly stops at the first handful of findings, or skips the re-check to save time, reads afterwards as if it covered everything when it didn't; if you bound the work, say what you left out. And adding agents to mask a bad brief — a critic loop on bad input produces critiqued bad input. Sharpen the brief before reaching for a second agent.

The decision rule. The first move is almost always to improve the brief. A second agent on a bad brief produces a second bad result. Add an agent only when one of three conditions holds. Independent perspective: the work needs a viewpoint the original brief can't structurally provide. Real parallelism: sub-tasks are genuinely independent and wall-clock time matters. Or cost of failure greatly exceeds cost of the extra agent: a second-opinion call on an irreversible decision costs one model call; the alternative may cost much more. If none of those holds, you need a better brief, not another agent.

## Chapter nine — Tool discipline, pick the right hammer

Every tool call has a cost — latency, tokens, context window space, sometimes a permission prompt — and a value, which is the information or change it produces. Tool discipline is the habit of reaching for the cheapest tool that delivers the needed value, and of batching independent calls into a single turn so the cache stays warm and the wall clock stays short. Done well, this is invisible. The session finds what it needs quickly, changes only what it should, and leaves the context holding distilled information rather than noise.

Same task, wrong tools: slower, noisier, more permission prompts, more context consumed. That context is finite — it has to hold your reasoning, not raw file dumps.

The first rule is: prefer dedicated file tools over their shell equivalents. For reading a file, use the Read tool, not the shell command "cat". For editing, use Edit, not "sed". For creating or overwriting, use Write, not "echo greater-than file". For listing files by pattern, use Glob, not "find". For searching file contents, use Grep, not shell grep. The dedicated tools render line-numbered output, produce structured diffs, avoid most permission prompts, and refuse ambiguous edits. They're auditable, reviewable, and safer than the shell equivalents. Save the shell for genuine subprocess work — running a build, querying a live system, restarting a service.

The second rule is: issue parallel calls when work is independent. If two or more tool calls don't depend on each other's results, issue them in the same turn. The runtime is bounded by the slowest call, not the sum. Three independent searches done in parallel finish in the time of the slowest. Three done in sequence finish in the sum of all three. The serial version is pure dead time. "To be safe" is not safe; it's just slow.

The third rule is: route large outputs through a subagent. When a tool call would return a large payload — a full log, a directory listing, a bulk query result — ask whether you actually need all of it in the main context. If what you need is a summary, a count, or the answer to a specific question, delegate the retrieval *and* the interpretation to a subagent. The subagent does the heavy reading; you get back two hundred words instead of twenty-five kilobytes.

The fourth rule is about hosts, not tools. If you have a fast-feedback orchestrator host and a separate heavy-compute executor, long-running work belongs on the executor. Builds, test suites, big data jobs — none of that should run on the host you're trying to keep responsive. The symptom of getting this wrong is "everything feels slow" even though the heavy machine is idle. The cause is running heavy work on the interactive host.

There's a secondary point about deferred tools. Most tools aren't loaded by default; you have to fetch them with a ToolSearch call. Before reaching for ToolSearch to load a single tool, ask whether there's a logical bundle. A single search that names a toolkit returns the whole set in one round-trip. Loading tools one at a time is wasted overhead; loading more than you need wastes a different kind of overhead. Load the bundle for the domain you're entering.

When working with external systems, there's a tier of options. Best is a dedicated integration — what Anthropic calls a "Model Context Protocol" server, an MCP, purpose-built for that system. Reach for that first. If none exists, drive a browser through a browser MCP. As a last resort, use computer-use — interpreting pixels on a desktop. Computer-use is fragile and slow; pixel-based interaction breaks on layout changes. Reach down the tiers only when the upper tiers are unavailable.

Here's how this plays out. A historian is working through several hundred digitised field notes spread across thematic subdirectories. She needs to find every reference to a specific surveying instrument, read the relevant passages, then edit a draft essay to replace an outdated name. Her first instinct: open files one by one. She "cats" the first directory's listings, reads ten files in sequence, finds two references. She's ten percent through and her context is full of files that contained nothing useful. She resets. One Grep call across the entire archive — recursive, with line numbers and context. Twenty-three matches in fourteen files, returned in a few hundred tokens. Then fourteen parallel Read calls scoped to a narrow line range around each match. Her context now holds exactly the passages she needs. Finally, one Edit with "replace all" to rename the instrument throughout her essay: one call, no regex, a clean diff. Three turns instead of several dozen. The context holds evidence and reasoning, not hundreds of lines of irrelevant material.

One more discipline belongs in this chapter, and it's about what you ask the assistant to *build*, not just what it reaches for in a session. An assistant this capable will cheerfully build you a custom dashboard for anything — and the dashboard is great for the week after it ships. Then something moves, the hand-maintained list behind it lags reality, and it quietly stops being opened. The failure isn't the technology; it's that it's bespoke at all. Before commissioning a status surface or an operations tool, ask what you would honestly open every day — and if the answer maps to an existing, actively-maintained tool, adopt that instead. Build custom only when nothing adopted covers the need, and the result can derive its data live from a source of truth that already exists, rather than from a list someone has to keep updating.

The short rule of thumb: use the most specific tool that does the job. Batch every independent call into the same turn. Route large outputs through a subagent and bring back only the derived result. Load deferred tool bundles once, not one tool at a time.

# Part four — Putting it together

## A walked example: Alex and the consultancy

Alex runs a one-person IT consultancy serving six small manufacturing clients. Different cloud providers, different on-premise hardware, different support agreements. Recurring monthly health checks, periodic security reviews, change requests, occasional incidents. Alex is simultaneously the strategist, the engineer, the documentarian, and the account manager. A general-purpose chat assistant that starts at zero every session is not a useful tool here. A properly built setup is.

Alex's **memory store** has about twenty entries across the four types. Two user entries capture standing preferences. Six project entries, one per client, hold infrastructure overview, config repository location, decisions already made, known quirks. A growing collection of feedback entries documents lessons hard-won — why one client's firewall silently drops health checks, that another's change approval requires forty-eight hours' notice. Several reference entries hold vendor API quirks.

Alex's **skills** are short and focused. A "slash monthly health check" skill takes a client name, reads the client's project memory file, runs the check sequence, and produces a formatted report. A "slash change request" skill drafts a scope document. A "slash incident triage" skill opens with structured symptom gathering. Each skill reads memory at invocation, so one definition serves all six clients.

Alex's **subagents** include a security-analyst agent — tools Read and Write only — that reviews configuration files against a hardening checklist. A documentation-drafter that handles heavy writing. When a security review covers three independent subsystems, Alex fans out three analysts in parallel.

Alex's **hooks** include a Pre Tool Use hook on Write and Edit that blocks anything pointed at a client's production configuration directory unless the assistant first emits a clear confirmation. A Session Start hook injects today's date and checks the overnight inbox. A Stop hook fires a Telegram message when a long session finishes.

Alex's **CLAUDE-dot-em-dee** is a user-wide file with cross-client conventions — house tone for client-facing reports, the rule that hostnames are never written in plain text in outputs that might be archived externally. A project-scope file at the root of each client's working directory captures the interpretive frame for that client: a two-line glossary, the standing constraints — for example, "change windows are Friday nine pm to Saturday six am only" — and pointers to the relevant memory file.

Alex's **persistence ladder**: in-session todos for today's health check. A plan file in the plans folder for a six-week multi-phase upgrade. Memory for the certificate-rotation procedure that turns out to apply across four clients. Worktrees for risky architectural changes that might not work.

Alex's **async layer**: health checks for all six clients run as background subagents on Friday afternoons. Each sends a "starting" notification and a completion summary. If a check encounters something needing a decision, it pauses and alerts immediately. A phone-side webhook appends messages to an inbox file that the next session pops first.

Alex's **multi-agent patterns**: a critic agent reviews draft security reports before they go to a client. A second-opinion agent runs on high-stakes change-request scoping when rollback complexity is significant.

Alex's **tool discipline**: parallel Grep calls when looking for service names across config repositories. Read scoped to line ranges, not whole-file dumps. Edit for templates, not shell sed. Vendor MCPs loaded as bundles. Heavy work pushed to the executor host.

### Where to start, by horizon

In week one, the realistic goal is just to stop starting every session at zero. Add one user memory file. Add one project memory file. Write a ten-line CLAUDE-dot-em-dee. Wire up a Session Start hook that injects today's date. That's it.

In month one, the goal is to stop paying the repetition tax. Build three skills for the most-repeated procedures. Add a feedback memory entry after the first preventable mistake. Write a plan file for any work spanning sessions. Define one subagent for a category of heavy work.

In quarter one, the goal is to let work happen without you. Add a protective hook on a path that must not be touched. Wire one outbound notification channel. Run a background job end-to-end. Trim the memory index.

In year one, you'll have a dozen memory entries (mostly feedback), six to ten skills, two or three subagent definitions, hooks that have caught real problems, and multi-agent critic loops on at-stakes deliverables. A working environment that compounds instead of resetting.

### When not to bother

The patterns in this guide pay off when work compounds — when you'll be in the same project tomorrow, next week, next month. When the same procedures recur. When the same mistakes can keep happening. They don't pay off for one-off questions, throwaway scripts, or research you'll never revisit.

Don't build a memory store for a project you'll finish this week. Don't write a skill for something you'll do once. Don't write a hook for a rule no one would ever break anyway. Don't add an agent when sharpening the brief would fix the problem. The system serves the work, not the other way around.

## Closing

Mastery of Claude Code is not about better prompts. It's about choosing the right durable structure for each piece of knowledge, procedure, expert, and guardrail your work needs. A cleverly worded instruction in a chat window is fragile. The same intent expressed as a memory file, a skill, a subagent brief, or a hook is robust. Over weeks and months, a well-structured setup improves; a prompt-engineering practice degrades, because the gap between what you remember to say and what the session needs to know keeps widening.

Build incrementally. Add the next pillar when you feel the friction it solves. The goal isn't to have all nine pillars running — it's to have the right pillars running for the work you're doing today, and a clear path to add the next one when the work calls for it.
